当一个需求只改一个仓库时，Git 分支、状态和目录通常还能靠记忆管理。真正麻烦的是另一种日常：前端仓库要改，后端接口要联调，小程序也要补一处展示，最后还要记得切回主分支、拉更新、清理临时 worktree。

Visual Worktree 解决的就是这种多仓库任务开发的碎片感。它把本地多个 Git 仓库扫描出来，再按「任务」聚合 worktree，让你看到一个需求背后实际牵涉的全部项目。

源码仓库：[imberZsk/visual-worktree](https://github.com/imberZsk/visual-worktree)

## 一个任务，一组仓库

Visual Worktree 的核心组织方式是：

```text
worktreesRoot/{任务名}/{项目名}
```

这意味着一个需求可以拥有自己的任务目录，目录下每个项目都是独立 worktree。你不用在多个源仓库之间反复切分支，也不用担心一个仓库的临时需求分支打断另一个任务。

![Visual Worktree 任务视图](/img/visual-worktree-overview.png)

在任务视图里，应用会把同一任务下的项目聚合展示。每个任务可以标记研发阶段、挂需求链接、查看流程进度，也可以快速打开 Finder、VSCode、终端或项目 GitLab 页面。

## 从状态巡检到批量动作

Visual Worktree 会扫描源项目目录，并展示每个仓库的 Git 状态：

- 当前分支是否为主分支
- 工作区是否有未提交变更
- 是否存在未推送提交
- 是否落后远程分支
- 该项目是否关联了 worktree

这些状态不只用于展示。选中多个项目后，可以批量切回主分支、批量拉取更新或批量暂存变更。批量执行时会逐项上报进度，单个项目失败不会阻断其他项目，这对几十个仓库的工作区整理尤其重要。

## 流程、卡点和环境都留在任务上

跨仓库需求不只是代码目录问题，还包含流程问题。Visual Worktree 给每个任务提供了一组可配置步骤，比如需求确认、开发、自测、提测、发布检查。步骤可以单纯打勾，也可以配置命令，在任务目录下执行脚本。

任务上还能记录卡点备注和外部链接。比如 Jira、飞书需求、测试用例、上线单都可以挂到同一个任务上。这样再次回到这个任务时，不需要从聊天记录里翻上下文。

环境健康检查则关注另一个常见痛点：任务目录能不能跑起来。应用会检查依赖、端口、服务连通性和 Git 状态，把环境问题直接展示在任务行上。

## 本地优先，不碰你的仓库内容

Visual Worktree 是本地桌面工具。配置、任务状态、链接、流程勾选、环境检查缓存都默认保存在：

```text
~/.visualWorktree
```

应用会读取你配置的本地仓库目录和 Git 状态，但不会把仓库内容、任务状态或本地路径上传到远端服务。这一点对内部研发工具很重要：它应该帮你整理工作台，而不是把工作台变成新的数据出口。

## 适合什么时候使用

Visual Worktree 特别适合这些场景：

- 一个需求经常要跨多个仓库一起改
- 本地存在很多长期维护的项目目录
- 需要频繁检查分支、未提交变更和远程差异
- 希望按任务维度跟踪研发流程、环境状态和外部链接
- 想保留 Git worktree 的灵活性，但需要更直观的桌面视图

它不是 Git 客户端的替代品，而是一个围绕「多仓库任务开发」重新组织的工作台。Git 继续负责版本控制，Visual Worktree 负责把分散的状态摆到你能一眼看懂的位置。
